# Firestore

Casa de Historias uses **Cloud Firestore** as its primary database for storing stories and user data.

## Data Structure

Firestore uses a NoSQL document-based structure:

```
casa-de-historias/
├── stories/ (collection)
│   ├── {storyId}/ (document)
│   │   ├── id: string
│   │   ├── titleEs: string
│   │   ├── titleNahuatl: string
│   │   ├── contentEs: string
│   │   ├── contentNahuatl: string
│   │   ├── audioUrl: string?
│   │   ├── imageUrl: string?
│   │   ├── narratorName: string
│   │   ├── community: string
│   │   ├── latitude: number?
│   │   ├── longitude: number?
│   │   ├── authorId: string
│   │   ├── tags: array<string>
│   │   ├── published: boolean
│   │   ├── createdAt: timestamp
│   │   └── updatedAt: timestamp
│   └── ...
└── users/ (collection)
    ├── {userId}/ (document)
    │   ├── id: string
    │   ├── email: string
    │   ├── displayName: string
    │   ├── photoUrl: string?
    │   ├── community: string
    │   ├── isStoryteller: boolean
    │   ├── favoriteStories: array<string>
    │   ├── createdAt: timestamp
    │   └── lastLogin: timestamp
    └── ...
```

## Collections

## Queries

### Get All Published Stories

```kotlin
suspend fun getAllStories(): List<StoryFirestore> {
    val snapshot = firestore.collection("stories")
        .whereEqualTo("published", true)
        .orderBy("createdAt", Query.Direction.DESCENDING)
        .get()
        .await()

    return snapshot.documents.mapNotNull { doc ->
        doc.toObject(StoryFirestore::class.java)
    }
}
```

See `data/firebase/FirestoreRepository.kt:39`.

**Query breakdown:**
- `.collection("stories")` - Target stories collection
- `.whereEqualTo("published", true)` - Filter published stories only
- `.orderBy("createdAt", Query.Direction.DESCENDING)` - Sort newest first
- `.get().await()` - Execute query and wait for result

### Get Stories by IDs (Batch)

```kotlin
suspend fun getStoriesByIds(ids: List<String>): List<StoryFirestore> {
    if (ids.isEmpty()) return emptyList()
    return try {
        // Firestore 'in' queries support max 30 items
        val results = mutableListOf<StoryFirestore>()
        ids.chunked(30).forEach { chunk ->
            val snapshot = firestore.collection("stories")
                .whereIn("__name__", chunk.map { firestore.collection("stories").document(it) })
                .get()
                .await()
            results.addAll(snapshot.documents.mapNotNull { doc ->
                doc.toObject(StoryFirestore::class.java)
            })
        }
        results
    } catch (e: Exception) {
        // Fallback: fetch one by one
        ids.mapNotNull { getStoryById(it) }
    }
}
```

See `data/firebase/FirestoreRepository.kt:97`.

This is used for fetching favorite stories. Firestore's `whereIn` has a limit of 30 items, so we chunk larger requests.

## Real-time Listeners

Firestore supports real-time updates via snapshot listeners.

### Real-time Stories Flow

```kotlin
fun getAllStoriesFlow(): Flow<List<StoryFirestore>> = callbackFlow {
    val listener = firestore.collection("stories")
        .whereEqualTo("published", true)
        .orderBy("createdAt", Query.Direction.DESCENDING)
        .addSnapshotListener { snapshot, error ->
            if (error != null) {
                close(error)
                return@addSnapshotListener
            }

            val stories = snapshot?.documents?.mapNotNull { doc ->
                doc.toObject(StoryFirestore::class.java)
            } ?: emptyList()

            trySend(stories)
        }

    awaitClose { listener.remove() }
}
```

See `data/firebase/FirestoreRepository.kt:19`.

**How it works:**
1. Creates a Firestore snapshot listener
2. Wraps it in a Kotlin `callbackFlow`
3. Sends updates whenever data changes
4. Automatically cleans up listener when Flow is closed

**Benefits:**
- UI automatically updates when stories are added/modified
- No manual refresh needed
- Efficient - only changed documents are transmitted

## Write Operations

### Add Story

```kotlin
suspend fun addStory(story: StoryFirestore): String? {
    return try {
        val documentRef = firestore.collection("stories").document()
        val storyWithId = story.copy(
            id = documentRef.id,
            createdAt = Timestamp.now(),
            updatedAt = Timestamp.now()
        )
        documentRef.set(storyWithId).await()
        documentRef.id
    } catch (e: Exception) {
        null
    }
}
```

See `data/firebase/FirestoreRepository.kt:118`.

**Process:**
1. Generate new document reference with auto ID
2. Copy story with generated ID and timestamps
3. Write to Firestore
4. Return document ID (or null on failure)

### Update Story

```kotlin
suspend fun updateStory(story: StoryFirestore): Boolean {
    return try {
        firestore.collection("stories")
            .document(story.id)
            .set(story.copy(updatedAt = Timestamp.now()))
            .await()
        true
    } catch (e: Exception) {
        false
    }
}
```

See `data/firebase/FirestoreRepository.kt:133`.

**Note**: Uses `.set()` instead of `.update()` to replace the entire document. The `updatedAt` timestamp is refreshed.

## Indexing

For queries with multiple filters and orderBy, create composite indexes:

### Required Indexes

1. **Stories by community (published)**
   - Collection: `stories`
   - Fields: `community` (Ascending), `published` (Ascending), `createdAt` (Descending)

2. **Stories by author**
   - Collection: `stories`
   - Fields: `authorId` (Ascending), `createdAt` (Descending)

Firebase will prompt you to create these indexes when you first run queries that need them.

## Search Limitations

Firestore doesn't support full-text search, so the app uses **client-side filtering**:

```kotlin
suspend fun searchStories(query: String): List<StoryFirestore> {
    // Fetch all published stories
    val allStories = firestore.collection("stories")
        .whereEqualTo("published", true)
        .get()
        .await()
        .documents.mapNotNull { it.toObject(StoryFirestore::class.java) }

    // Filter locally
    val lowerQuery = query.lowercase()
    return allStories.filter { story ->
        story.titleEs.lowercase().contains(lowerQuery) ||
        story.titleNahuatl.lowercase().contains(lowerQuery) ||
        story.narratorName.lowercase().contains(lowerQuery) ||
        story.community.lowercase().contains(lowerQuery) ||
        story.tags.any { it.lowercase().contains(lowerQuery) }
    }
}
```

See `data/firebase/FirestoreRepository.kt:154`.

**For production**, consider:
- Using Algolia or Elasticsearch for full-text search
- Implementing Cloud Functions to index content
- Caching all stories locally for instant search

## Best Practices

1. **Always filter by `published: true`** for public queries
2. **Use `orderBy` for consistent results**
3. **Limit large queries** with `.limit()`
4. **Handle errors gracefully** with try-catch
5. **Use real-time listeners** for data that updates frequently
6. **Batch reads** when fetching multiple documents by ID
7. **Update `updatedAt`** timestamp on every write
8. **Use `FieldValue`** for array and increment operations
